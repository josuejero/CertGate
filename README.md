# CertGate Validation Gate

## Why CertGate
- **Automated release guardrail** for candidate, exam, and certification feeds with data-quality decisions baked into every pipeline run.
- **Severity-aware gating** links schema, integrity, and freshness checks to explicit release outcomes (`Ready`, `Warning only`, `Blocked`).
- **Recruiter-ready showcase**: interactive demo site, Great Expectations data docs, and structured `reports/*.json` artifacts explain what happened, when, and why.

## Quick Highlights
- Guards every ingest using pandas helpers (`src/rules/schema_rules.py`, `src/rules/business_rules.py`) before Great Expectations milestones execute.
- Reports validation results, defects, and decisions via `src/reporting.py` to create `reports/validation_summary.json`, `reports/defect_summary.json`, and `reports/release_decision.json`.
- Demo site in `demo-site/` references those artifacts, plus automated screenshots/GIFs generated by `scripts/generate_demo_assets.py`.

## Data Feeds & Schemas
| Feed | Required columns | Uniqueness constraints | Severity implications |
| --- | --- | --- | --- |
| `candidates.csv` | `candidate_id`, `birth_date`, `email`, `first_name`, `last_name`, `created_ts`, `source_system` | `candidate_id`, `email` | Missing columns or duplicates block release (`critical`). See `docs/business-rules.md` for details. |
| `exam_results.csv` | `exam_result_id`, `candidate_id`, `exam_code`, `exam_date`, `attempt_number`, `score`, `score_scaled`, `pass_fail`, `exam_location`, `file_received_ts` | `exam_result_id`; (`candidate_id`, `exam_code`, `attempt_number`) | Invalid scores, stale exams, or duplicate attempts produce `warning`/`critical` outcomes depending on the rule (`docs/rule-severity-matrix.md`). |
| `certification_status.csv` | `cert_status_id`, `candidate_id`, `certification_level`, `certification_status`, `certified_ts`, `status_source`, `status_effective_ts` | `cert_status_id`; (`candidate_id`, `certification_level`) | Certification mismatches or overlapping statuses raise `warning`/`critical` defects in `docs/rule-severity-matrix.md`. |

## Validation Architecture
- **Schema helpers:** `src/rules/schema_rules.py` defines `SchemaDefinition` objects, column expectations, dtype normalization, and uniqueness detection. It emits `RuleOutcome` dataclasses that describe pass/fail status, severity (`critical`/`warning`/`info`), and metadata for downstream reporting.
- **Business logic:** `src/rules/business_rules.py` adds referential integrity (`BR-07`, `BR-07b`), pass/fail-to-certification mapping (`BR-05`), and freshness analysis (`BR-08`). Each helper returns `RuleOutcome` objects consumed by `ReleaseReport`.
- **Reporting:** `src/reporting.py` aggregates all outcomes, infers root causes, and writes the `reports/` artifacts (`validation_summary.json`, `defect_summary.json`, `release_decision.json`). These payloads drive automated awareness (e.g., release gates, demos) and link back to `docs/rule-severity-matrix.md` for remediation.

## Automation & Reporting Scripts
| Script | Purpose | Sample command |
| --- | --- | --- |
| `python scripts/run_great_expectations.py` | Executes GX checkpoint `ingest_validation_checkpoint` against canonical `data/good/*.csv` bundles; populates `gx/validations/` and publishes docs via `gx/data_docs/`. | `python scripts/run_great_expectations.py` |
| `python scripts/generate_release_reports.py` | Loads every schema via `src/ingest/loaders.py`, runs schema/business rule checks, and dumps `reports/` exports via `ReleaseReport`. | `python scripts/generate_release_reports.py` |
| `python scripts/generate_demo_assets.py` | Creates hero screenshots + animated workflow GIF used by `demo-site/`. | `python scripts/generate_demo_assets.py` |
- **Artifacts:** `reports/release_decision.json` is the single-source gate status, while `reports/validation_summary.json` and `reports/defect_summary.json` enumerate every evaluated rule plus root causes (schema/integrity/freshness/duplicates/business-rule drift). The `demo-site/` reads these files at runtime, so any script rerun refreshes the showcase.
- **Great Expectations docs:** After `run_great_expectations.py`, `gx/data_docs/index.html` and suite-specific pages (`candidates_suite`, `exam_results_suite`, `cert_status_suite`, `freshness_suite`) become inspectable via browser.

## Testing & QA Layers
- **Unit tests** (`tests/unit/`) verify helper logic for `schema_rules`, `business_rules`, and `reporting` using parametrized fixtures and synthetic slices from `data/`.
- **Functional/regression/uat layers** (folders under `tests/functional`, `tests/regression`, `tests/uat`) exercise multi-file flows, failure modes, and integration with `Great Expectations` checkpoints. Each layer runs against tailored datasets (`data/good/`, `data/bad/`, `data/regression/`) described in `docs/test-plan.md`.
- **Synthetic datasets:** `data/bad/` contains curated failure cases (duplicate candidates, missing candidate join, future exam dates, etc.), while `data/regression/` captures multi-feed interactions to stress cross-file rules. `docs/defect-log.md` chronicles recurring failures and remediation history.
- **Artifacts & CI:** Every test run emits `reports/junit.xml`, `reports/validation_summary.json`, and `reports/release_decision.json`. Packaging these artifacts allows downstream workflows (CI, demo site, release automation) to assert severity levels against the rule severity matrix in `docs/rule-severity-matrix.md`.

## Demo & Stakeholder Storytelling
- The static `demo-site/` mirrors what a recruiter sees: status pills, rule counts, artifact links, and a defect table backed by real report JSON files. Visitors can click through the Great Expectations data docs and download release artifacts directly.
- Visual assets include failures/passes (`demo-site/assets/*.png`) plus an animated narrative GIF (`workflow.gif`) generated by `scripts/generate_demo_assets.py`. Use these in presentations to show how the gating cycle moves from blocked runs to ready releases.

## Getting Started
1. **Install dependencies:** <br>`pip install -r requirements.txt`<br>`pip install great_expectations` (versions locked per `requirements.txt`).
2. **Run schema + business guards:** <br>`python scripts/generate_release_reports.py` produces `reports/*.json` capturing every rule outcome.
3. **Run Great Expectations checkpoint:** <br>`python scripts/run_great_expectations.py` (requires `great_expectations` config in `gx/`).
4. **Refresh demo assets (optional):** <br>`python scripts/generate_demo_assets.py` regenerates hero visuals for the demo site.
5. **Inspect outputs:** <br>`demo-site/index.html` loads `reports/*.json`; `gx/data_docs/index.html` exposes validation results; `docs/rule-severity-matrix.md` explains severity/resolution steps; `docs/business-rules.md` and `docs/test-plan.md` provide deeper domain context.
